# ADR-003 — WorkManager 동기화 주기 (30분 PeriodicWork)

- **상태**: Accepted
- **작성일**: 2026-04-25 (Week 6)
- **결정자**: lsk12
- **연관**: 명세서 §3 Week 6, §7 ② Offline-first

## 1. 컨텍스트

걸음수를 서버로 동기화하는 주기 결정. 너무 짧으면 배터리, 너무 길면 사용자가 다른 기기에서 랭킹 볼 때 데이터 stale.

## 2. 고려한 옵션

### 옵션 A — 15분 (PeriodicWork 최소)
- 장점: 거의 실시간
- 단점: 배터리 부담, Doze 영향 있음

### 옵션 B — 30분
- 장점: 배터리 부담 감소, Doze 시 영향 최소
- 단점: 사용자가 직전 30분 걸음수는 다른 기기에서 못 봄

### 옵션 C — 1시간
- 장점: 배터리 최소
- 단점: 그룹 랭킹 갱신감 떨어짐

### 옵션 D — Adaptive (산책 종료 시 즉시 + 정기)
- 장점: UX ↑
- 단점: 복잡도 ↑

## 3. 결정

**옵션 B (30분)** 채택. 산책 종료 시 즉시 동기화는 v1.1 검토.

## 4. 근거

- WorkManager `PeriodicWork` 최소 단위 15분이지만, Doze에서 변동 폭 큼
- 30분이 배터리/UX 균형점

## 5. 결과

### 긍정적
- 배터리 영향 최소
- 면접 답변 ② Offline-first 스토리에 활용

### 부정적
- 직전 30분 걸음수 stale (단, 자기 기기에서 보면 즉시 보임)

### 재검토 트리거
- 사용자 피드백에서 "랭킹 늦다" 다수 → 산책 종료 트리거 즉시 동기화 추가
